# How the Search Engine Thinks

A plain-English walkthrough of every step — from the moment a user types a query
to the moment results appear on screen.

---

## The Big Picture

```
User types a query
       ↓
  Step 1 — Alias Check         "12v car plugin" → "car charger"
       ↓
  Step 2 — Rule Engine          Is this a part number? A laptop? An accessory?
       ↓
  Step 3 — AI Interpretation    Expand & clarify the intent
       ↓
  Step 4 — Database Search      Original query first, then AI expansions (up to 60 results each)
       ↓
  Step 5 — Scoring              Field weights + model number match + stock adjustment
       ↓
  Step 6 — Consumer Tier Boost  +8 for everyday consumer product categories
       ↓
  Step 7 — Category Boost       +50 in-category / −25 out-of-category
       ↓
  Step 8 — Exclude Filter       Drop results whose category contains an excluded term
       ↓
  Step 9 — Image Fetch          Thumbnails pulled from image databases via TocAN join
       ↓
  Top 20 results returned
```

---

## Step 1 — Alias Check (keyword translation)

**What it does:** Before anything else, the engine checks if the user typed a
phrase that should be translated into a better search term.

**Where the aliases come from:** The admin panel → "Keyword Aliases" tab, or
populated automatically by running `populate_aliases.py`, or generated by the
AI alias generator (admin → "Generate Aliases" tab — searches Amazon, Alibaba
and AliExpress for real listings and extracts buyer language).

**Example:**
```
User types:    "12v car plugin"
Alias table:   "12v car plugin" → "car charger"
Engine uses:   "car charger"  ← this is what actually gets searched
```

Matching goes **longest alias first** so "cigarette lighter plug" matches
before "lighter" alone. Only the first matching alias is applied.

If no alias matches, the original query is used as-is.

---

## Step 2 — Rule Engine (fast classification, no AI)

The rule engine looks at the (possibly aliased) query and tries to classify it
using simple logic — no AI involved, happens in milliseconds.

### 2a. Is it a part number?

The engine checks if the query looks like an electronic component code.

**Triggers:**
- Starts with known IC prefixes: `LM`, `NE`, `TL`, `TIP`, `BC`, `74`, `40`...
- Mix of letters and digits in a single word: `LM358`, `BC547`, `74HC595`
- Any single word containing both letters and numbers

**Result:** Searches all fields exactly as typed. No category boost. AI is skipped entirely.

```
"lm358"     → part number search
"arduino"   → NOT a part number (no digits)
"LM7805"    → part number search
```

### 2b. Is it a product + accessory modifier?

The engine checks if the query contains an accessory word alongside a product name.

**Accessory words that trigger this:**
`bag`, `case`, `cover`, `charger`, `adapter`, `cable`, `fan`, `battery`,
`screen protector`, `keyboard`, `stand`, `dock`, `hub`, `socket`, `jack`, `connector`

**Rule:** If the query has MORE than one word AND contains an accessory modifier,
the engine treats it as a specific accessory search.

```
"dell charger"    → accessory search → filter to "Adapters" category
"hp battery"      → accessory search → filter to "Laptop Batteries" category
"laptop bag"      → accessory search → filter to "Bags & Carry Case"
"laptop"          → NOT triggered (only one word, no modifier)
```

### 2c. Is it a known product type?

The engine checks the **Product Terms table** (managed in admin → "Product Terms"
tab, or populated by `populate_terms.py`).

This table has entries like:

| Term       | Category Boost        | Exclude Terms                                  |
|------------|-----------------------|------------------------------------------------|
| laptop     | Laptops               | socket, jack, adapter, bag, fan, battery, ...  |
| macbook    | Laptops               | socket, jack, adapter, bag, fan, battery, ...  |
| iphone case| Cases & Covers        |                                                |
| router     | Routers               |                                                |
| arduino    | Arduino Microcontrollers |                                             |

**Rule:** If the query contains a known product term AND does NOT contain an
accessory modifier, the engine marks this as a "product type" search.

```
"acer laptop"  → "laptop" found in table → boost to "Laptops"
"dell"         → not in table → falls through to "general"
"ipad case"    → "case" is an accessory modifier → overrides the ipad rule
```

### 2d. General search (nothing matched)

If none of the above matched, the query is marked as a general search.
Everything is searched, no category filters, no boosts.

---

## Step 3 — AI Interpretation

The rule engine's classification is sent to an AI model (fast model) to expand
and refine the search. Skipped entirely for part number queries.

**What the AI receives:**
- The user's query
- The rule engine's decision ("this is a laptop search", "boost to Laptops", etc.)
- A description of the inventory
- A priority rule: consumer products always surface before technical components
  when the query is ambiguous

**What the AI returns:**
```json
{
  "search_terms": ["acer laptop", "acer notebook", "acer aspire"],
  "exclude_terms": ["battery", "inverter"],
  "preferred_category": "Laptops",
  "interpretation_note": "User is looking for an Acer laptop computer"
}
```

The `search_terms` list drives multiple database passes — one per term — to
maximise coverage.

---

## Step 4 — Database Search

The engine runs up to 4 database passes:
1. **Original query always goes first** — raw user input before any AI expansion
2. Then each AI-suggested term, in order
3. Stops early once 20 unique results have been collected

Each pass fetches up to **60 candidates** from the database.

### What fields are searched

| Field             | Match type    | Why                                        |
|-------------------|---------------|--------------------------------------------|
| Description       | Word boundary | Main product name                          |
| DescriptionArabic | Word boundary | Arabic product name                        |
| ItemAnotherName   | Word boundary | Alternate names / tags                     |
| WebDescription    | Word boundary | Extended description                       |
| ItemNote          | Word boundary | Extra notes                                |
| Brand name        | Word boundary | Brand matching                             |
| Original_No       | Substring     | Part numbers / model numbers               |
| Item (SKU code)   | Substring     | Internal product codes                     |
| Barcode           | Substring     | Barcode scanning                           |

**Word boundary** means `"acer"` will NOT match inside `"spacer"` or `"tracer"`.
It must appear as a standalone word. This prevents false positives from
partial-word collisions.

**Substring** is used for codes because part numbers are often partial
(searching `"LM35"` should find `"LM35DZ"`).

### Filters applied in SQL

- Hidden products → excluded
- Deleted products → excluded
- Blocked products → excluded
- If a **category filter** was set → only that category is searched
- SQL pre-orders by: (1) does it match the target category? (2) is it in stock?

---

## Step 5 — Confidence Scoring

After fetching, each product gets a **confidence score** calculated in Python.

### Field weights

Each field that contains a search word adds points:

| Field           | Points |
|-----------------|--------|
| ItemAnotherName | 40     |
| Original_No     | 35     |
| Item (SKU)      | 30     |
| Description     | 25     |
| Brand           | 20     |
| CategoryName    | 15     |
| WebDescription  | 10     |
| ItemNote        |  5     |
| Barcode         |  5     |

**Match ratio multiplier:** If the query has 2 words and only 1 is found,
the score is multiplied by 0.5.

### Model number accuracy bonus/penalty

If the query contains numbers (e.g. `"iPhone 15"`, `"65W"`, `"XS Max"`):

| Situation                         | Adjustment |
|-----------------------------------|------------|
| All query numbers found in item   | **+25**    |
| Some query numbers found          | **+10**    |
| No query numbers found in item    | **−40**    |

This prevents e.g. an iPhone XS Max case appearing at the top when someone
searches for an iPhone 15 case.

### Stock adjustment

| Stock level      | Adjustment |
|------------------|------------|
| Balance > 0      | **+5**     |
| Balance = 0      | **−5**     |
| Balance < 0      | **−10**    |

Stock is intentionally a small factor — relevance always wins over availability.

**Example:**
```
Query: "acer laptop"

Product: "Acer Aspire 5 Laptop, Core i5"
  Description matched "acer" + "laptop" → +25
  Brand matched "acer"                  → +20
  Both words found (ratio = 1.0)        → score = 45 × 1.0 = 45
  No numbers in query → no model bonus
  In stock                              → +5
  Base score: 50
```

---

## Step 6 — Consumer Tier Boost

Before category boosting, products in everyday consumer categories get a
flat **+8 points** to ensure they surface above technical components when
the query is ambiguous.

Consumer categories include: Cases & Covers, Laptops, Earbuds & Headsets,
USB Cables, HDMI Cables, Power Banks, Bluetooth Speakers, Mobile Screen
Protectors, External Storage, Wireless Chargers, Smart Watches, Adapters,
Power Adapters, Routers, Monitors, Desktop Computers, and others.

---

## Step 7 — Category Boost

If the rule engine identified a **target category**, scores are adjusted:

- Products **in** the target category → **+50 points**
- Products **outside** the target category → **−25 points**

This is why `"acer laptop"` surfaces actual Acer laptops (in "Laptops") above
Acer laptop batteries (in "Laptop Batteries"), even if the batteries match
both words textually.

---

## Step 8 — Exclude Filter

After scoring and boosting, certain results are removed entirely.

Exclude terms come from:
1. The **Product Terms table** (e.g. "laptop" excludes categories containing
   "battery", "inverter", "backlight", "socket", "jack"...)
2. The AI interpretation (if the AI is confident certain categories are wrong)

**How it works:** If any exclude term appears inside a product's **CategoryName**,
that product is dropped.

```
Query: "laptop"
Exclude terms: ["battery", "inverter", "backlight"]

"Laptop Batteries"   → CategoryName contains "battery"  → REMOVED
"Laptop Inverters"   → CategoryName contains "inverter" → REMOVED
"Laptop Computers"   → no match                         → KEPT
"Laptop Fans"        → no match                         → KEPT
```

---

## Step 9 — Image Fetch

After ranking, product thumbnails are fetched in a single bulk query per image
database using a `TocAN` join (TocAN links image records to product AutoNums).

- Images found → embedded as base64 data URIs (no extra HTTP request on load)
- Images not found → fallback to `/api/image/{autonum}` lazy-loaded endpoint

---

## Final Ranking

Results are sorted by confidence score (highest first). The **top 20** are
returned to the frontend.

---

## When Nothing Is Found

If the search returns zero results, the engine does two things in parallel:

1. **Substitute finder** — queries the database for in-stock products in
   related categories and sends them to an AI model (smart model) to pick the
   best functional replacements. Returns up to 3 substitutes with a reason for
   each.

2. **Search suggestions** — an AI model generates 6 alternative search terms
   the user could try instead, shown as clickable chips.

---

## Image Search

When a user uploads a photo instead of typing:

1. The image is sent to an AI vision model
2. The AI detects brand, model number, and product type
3. Three search passes run in order, stopping at the first that finds results:
   - Pass 1: `"{brand} {model}"` — most specific
   - Pass 2: `"{brand}"` — brand only
   - Pass 3: `"{product_type}"` — generic fallback
4. The matched results go through the normal scoring pipeline from Step 5 onward
5. If nothing is found, substitute suggestions are generated using the product type

---

## Admin Controls

Everything about how the search behaves can be tuned without touching code:

| What you control                    | Where                              |
|-------------------------------------|------------------------------------|
| Keyword aliases (manual)            | Admin → Keyword Aliases            |
| AI-generated aliases from web       | Admin → Generate Aliases           |
| Product term → category rules       | Admin → Product Terms              |
| Category substitutions              | Admin → Category Substitutes       |
| Bulk-populate aliases from DB + AI  | Run `populate_aliases.py`          |
| Bulk-populate product terms         | Run `populate_terms.py`            |
